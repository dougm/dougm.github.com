Managed Object Description

Interface to manage the group for a replication target.

In the initial release, a consistency group can only contain a single Virtual Machine. In the future, it may include more than one VM.

Each group may have zero or more "consistent" instances (see latestInstances).

A group may have at most one image, no matter how many consistent instances it has. To find the image, check activeImage for this Group.

HbrReplicationGroupCommitToImage_Task(commitToImage)

Commit this replica to the given fail-over image, and disable this replication configuration. All files associated with the replica group, except those needed by the given image, will be cleaned up. The disk chain for the instances newer than the selected instance is discarded.

The files associated with the image (directly or indirectly via the disk hierarchy) will no longer be managed by this server.

Only "fail-over" images can be committed to. It is an error to commit to a test-bubble image. See HbrReplicationGroupCreateTestBubbleImage_Task and HbrReplicationGroupCreateFailoverImage_Task.

See BaseGroupRemove_Task which removes a group only if an existing image has been reverted, and see HbrReplicationGroupRevertImage_Task which removes an image without impacting the group state.

HbrReplicationGroupCreateFailoverImage_Task(createFailoverImage)

Create an image from an instance for this group in the given location.

The image config files are fixed up. (E.g., the datastore paths or disk labels, and the hbr replication configuration).

Only the config files for the virtual machine image are stored in the given location. The disks of the virtual machine are left where they were created during configuration of the replication.

Replication is blocked for the group before it starts to create the image files. In most common cases where a failover image is requested, the orchestrator managing VM recovery (such as vSphere Replication or VMware Site Recovery Manager) should already ensure that the protected VM at source site was powered off. Replication will continue when the failover image is reverted (HbrReplicationGroupRevertImage_Task).

When using the UDM replication style, the vmx will point to a dummy disk and will only be fixed to point at a valid disk when HbrReplicationGroupCommitToImage_Task is invoked. This allows the VM to be registered but it cannot be powered on until after a successful commit.

Caller must eventually HbrReplicationGroupRevertImage_Task or HbrReplicationGroupCommitToImage_Task the resulting image.

The Group cannot be destroyed until the image is removed or committed to.

If the disk mode of a replicated disk is independent-persistent, independent-nonpersistent, or nonpersistent, it is changed to be persistent.

This API does not apply if the group is idle. XXX new fault.

HbrReplicationGroupCreateTestBubbleImage_Task(createTestBubbleImage)

Create a test-bubble image from an instance of this group in the given location.

The image config files are fixed up. (E.g., the datastore paths or disk labels, and the hbr replication configuration).

The config files for the virtual machine image are stored in the given location. Additionally, redo logs for each disk of the virtual machine will be stored in the given location, too.

Caller must HbrReplicationGroupRevertImage_Task the resulting image to clean up disk state associated with the VM. The instance state this image is created from will exist until the image is destroyed. The Group cannot be destroyed until the image is reverted.

It is an error if the given directory already exists. Reverting a test-bubble image that is created in a directory with other 'useful' items in it is sketchy.

If the disk mode of a replicated disk is independent-persistent, independent-nonpersistent, or nonpersistent, it is changed to be persistent.

HbrReplicationGroupPauseRpoReporting(pauseRpoViolationReporting)

Pause the reporting for RPO violations.

No further property updates for #currentRpoViolation will be posted until HbrReplicationGroupResumeRpoReporting is called.

This is usually called after replication is paused at the primary side. This flag is persistent across hbrsrv reboot. This is an idempotent operation (it is a no-op when RPO violations reporting has already been paused).

See #currentRpoViolation which returns the current RPO violations.

This API does not apply if the group is idle. XXX new fault

HbrReplicationGroupResumeRpoReporting(resumeRpoViolationReporting)

Resume the reporting for RPO violations.

This function causes a property update with the current value for #currentRpoViolation to be posted (to ensure the caller has the correct value).

This is usually called after replication is resumed at the primary side. This flag is persistent across hbrsrv reboot. This is an idempotent operation (it is a no-op when RPO violations reporting has already been resumed).

See #currentRpoViolation which returns the current RPO violations.

HbrReplicationGroupUnconfigure_Task(unconfigure)

Prepare this group to be moved to another hbrsrv.

Like a BaseGroupRemove_Task except this does not clean up any files, leaving them behind with the expectation that a different HBR server will be tasked with picking up the pieces via the LoadReplicationGroup_Task method.

Note that a group with an associated image cannot be unconfigured. The image must be reverted before a group can be unconfigured.

Any replica disks that are unconfigured and ready for removal (i.e., they are not referenced by any instances) will be cleaned up (as if the VirtualMachine#removeDisk method were invoked on it) as a side-effect of the unconfigure.

Throws exceptions if the storage for the group is currently unreachable (and the state file cannot be written).
